# v0.3.0 Design: YAML Entrypoint & Config Singleton

**Date:** 2026-03-12
**Status:** Draft

## Overview

Add a YAML entrypoint feature to `configtree-loader`. At import time the library reads a YAML config file, loads any listed configtree directories, and exports a typed `config` singleton. Consumers get a single import for all runtime configuration.

## Requirements

- Read a YAML entrypoint file at startup (synchronous, at import time)
- Default path: `config/app.yaml` (relative to `process.cwd()`)
- Override via `APP_CONFIG_FILE` environment variable
- YAML file may contain a `configtrees` key (array of directory path strings) — these are loaded via the existing `loadConfigTreeSync`
- If `configtrees` is present but not an array, it is ignored (treated as absent)
- If `configtrees` is an array but contains non-string elements, those elements are silently skipped
- All declared configtree paths are mandatory — `loadConfigTreeSync` is called with default `{ optional: false }`, so missing directories throw
- YAML file may contain arbitrary other keys — these become top-level properties on `config`
- If the YAML file contains a top-level key `configtreeValues`, it is silently overwritten by the computed configtree result
- Loaded configtree file values are collected under `config.configtreeValues: Record<string, string>`
- `configtrees` key is stripped from the exported config
- Missing entrypoint file (`ENOENT`) is silent — returns `{ configtreeValues: {} }`
- Any other `readFileSync` error (e.g., `EISDIR` when `APP_CONFIG_FILE` points to a directory, permission denied) propagates as-is
- Invalid YAML throws (fatal — the app cannot start safely with malformed config)
- YAML top-level value that is not a mapping (e.g., bare string, array) throws
- `configtrees` paths are resolved to absolute paths via `resolve(process.cwd(), path)` (Node `path` module) before being passed to `loadConfigTreeSync`

## Side Effect on Import

Re-exporting `config` from `src/index.ts` means any consumer importing **anything** from `configtree-loader` will cause `src/config.ts` to be loaded, triggering the synchronous file read. This is a deliberate tradeoff for v0.3.0.

**Test isolation:** Importing `src/index.ts` in tests will trigger `_buildConfig`. This is harmless when `config/app.yaml` is absent (the `ENOENT` path is silent). However, if a CI environment has a stray `config/app.yaml` present in `process.cwd()` with invalid content, existing tests would throw. To prevent this, tests should set `APP_CONFIG_FILE` to a known-nonexistent path (e.g., via a `vitest.config.ts` env override) to guarantee the silent ENOENT path is always taken for non-config tests.

**Tree-shaking note:** `tsup.config.ts` has `treeshake: true`. Module-level statements without `/*#__PURE__*/` are not eliminated by rollup/tsup, so `_buildConfig(...)` executes whenever the bundle is loaded. Consumer-side bundlers that aggressively tree-shake may skip the initialization if `config` is unused — acceptable since Node.js (the primary target) does not bundle modules.

A `configtree-loader/config` sub-path export is tracked as a v0.4.0 follow-up for full isolation.

## Config Object Shape

```typescript
interface AppConfig {
  configtreeValues: Record<string, string>; // merged values from all configtree dirs
  [key: string]: unknown;                   // all other app.yaml top-level keys
}
```

**Library-internal note:** `tsconfig.json` enables `noPropertyAccessFromIndexSignature`. This applies only to the library's own source code. Within `src/config.ts` and tests, YAML keys accessed via the index signature must use bracket notation (e.g., `config["app"]`). The explicitly declared `configtreeValues` property is accessible via dot notation as normal. This does not affect downstream consumers.

Example `config/app.yaml`:

```yaml
configtrees:
  - ./config/db
  - ./config/secrets

app:
  name: my-service
  port: 8080
```

Resulting `config`:

```typescript
{
  app: { name: "my-service", port: 8080 },
  configtreeValues: {
    DATABASE_HOST: "localhost",   // from ./config/db/DATABASE_HOST
    DATABASE_PASS: "s3cret",      // from ./config/secrets/DATABASE_PASS
  }
}
```

## Merge Strategy

1. Parse YAML → assert result is a non-null, non-array object (throw if not)
2. Extract `configtrees` array (skip if absent or not an array); filter to string elements only
3. Strip `configtrees` key from the base object
4. For each path in `configtrees` (in order): resolve to absolute via `resolve(process.cwd(), path)`, then call `loadConfigTreeSync(absPath)` with default options (`{ optional: false }`) — one call per path for clear per-path error attribution; errors propagate unwrapped
5. Merge all configtree results into `configtreeValues` — later paths overwrite earlier for same key
6. Final object: `{ ...base, configtreeValues }` — computed `configtreeValues` always wins over any YAML key of the same name

## Type Narrowing

`yaml.parse()` returns `unknown`. `_buildConfig` narrows it before use:

```typescript
const parsed: unknown = parse(source);
if (typeof parsed !== "object" || parsed === null || Array.isArray(parsed)) {
  throw new Error("configtree-loader: app.yaml must be a YAML mapping");
}
const base = parsed as Record<string, unknown>;
const rawTrees: unknown = base["configtrees"];
const trees: string[] = Array.isArray(rawTrees)
  ? rawTrees.filter((p): p is string => typeof p === "string")
  : [];
```

This satisfies `noUncheckedIndexedAccess` without unsafe assertions.

## Architecture

### New file: `src/config.ts`

- Exports `AppConfig` interface
- Exports `_buildConfig(filePath: string): AppConfig` — pure function, testable. **Not re-exported from `src/index.ts`**; tests import it directly from `"../src/config.js"`. Since tsup only emits type declarations for symbols reachable from the entry point's re-exports, `_buildConfig` will not appear in `dist/index.d.ts`.
- Exports `config: AppConfig` singleton — calls `_buildConfig` once at module load

```typescript
export const config: AppConfig = _buildConfig(
  process.env["APP_CONFIG_FILE"] ?? resolve(process.cwd(), "config/app.yaml")
);
```

The env var wiring (`APP_CONFIG_FILE`) is module-level code and is not directly unit-tested. This gap is accepted: the wiring is a one-liner and `_buildConfig` covers all meaningful logic paths.

### Updated: `src/index.ts`

Adds re-exports (only public API — `_buildConfig` is not included):

```typescript
export { config, type AppConfig } from "./config.js";
```

### Build (`tsup.config.ts`)

No changes needed. `src/config.ts` is imported by `src/index.ts` and bundled inline automatically.

## Test Cases

New tests import `_buildConfig` directly from `"../src/config.js"`.

| Scenario | Expected |
| --- | --- |
| Missing yaml file | `{ configtreeValues: {} }` |
| Non-ENOENT readFileSync error (EISDIR) | propagates / throws |
| Valid yaml, no `configtrees` key | yaml values + `configtreeValues: {}` |
| Valid yaml with `configtrees` | yaml values + merged configtree values |
| Later configtree path overwrites earlier key | last write wins in `configtreeValues` |
| `configtrees` key stripped from output | key absent on `config` |
| `configtrees` is a scalar (not an array) | ignored, `configtreeValues: {}` |
| `configtrees` is an array with non-string elements | non-string elements skipped |
| YAML top-level `configtreeValues` key | silently overwritten by computed result |
| Invalid YAML | throws |
| YAML top-level is not a mapping (bare string, array) | throws |
| Missing declared configtree dir | throws (unwrapped from `loadConfigTreeSync`) |
